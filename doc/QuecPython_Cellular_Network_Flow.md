# QuecPython 蜂窝网络连接流程详解

本文档详细解析了 QuecPython 模组（如 EC800M）的蜂窝网络连接机制，以及 `network_manager.py` 在其中的作用。

## 1. 什么是蜂窝上网？

**蜂窝上网 (Cellular Networking)** 指的是通过运营商基站（移动、联通、电信）进行的移动数据通信。它类似于手机上网，需要 SIM 卡，并通过特定的信令流程与基站建立连接，获取 IP 地址。

核心概念：
*   **APN (Access Point Name)**: 接入点名称，相当于运营商的网关大门（如 `cmnet`, `3gnet`）。
*   **PDP Context (Packet Data Protocol Context)**: 数据包协议上下文，可以理解为“一条拨通的上网线路”。激活 PDP 上下文 = 拨号上网。
*   **Profile ID**: 配置文件 ID（通常为 1），用于存储 APN、用户名、密码等拨号参数。

## 2. 模组的默认行为：开机自动激活

在 QuecPython 固件中，默认行为是 **“开机自动拨号”**。

*   **流程**：
    1.  模组上电。
    2.  自动检测 SIM 卡。
    3.  自动搜索基站信号（注网）。
    4.  **自动尝试激活 Profile 1 (PDP Context 1)**。
    5.  如果激活成功，获取 IP 地址，此时用户代码（如 `main.py`）启动时，网络可能已经通了。

*   **问题**：
    *   如果 SIM 卡需要特定的 APN（比如物联网卡需要 `cmiot`），而模组默认使用通用 APN，自动拨号可能会失败。
    *   或者自动拨号虽然发起了，但用户代码运行太快，此时还没拨号成功，导致用户代码报错。

## 3. 我们做了什么？(`network_manager.py`)

在 `network_manager.py` 中，我们并没有改变“开机自动激活”这个底层机制（因为这是固件写死的），而是做了一层 **“智能等待与状态管理”**。

### 3.1 之前的尝试（V1.0 - 手动干预版）
我们在 V1.0 版本中尝试了：
1.  手动去检测运营商。
2.  手动设置 APN (`dataCall.setPdpContext`)。
3.  手动发起激活 (`dataCall.activate`)。

**结果**：失败。原因往往是模组已经在后台自动尝试拨号，我们的手动指令与后台自动流程冲突，导致资源占用或状态混乱（比如返回 `ret: -1`）。

### 3.2 现在的方案（V2.0 - 顺势而为版）
我们在 V2.0 版本中选择了最稳妥的方案：**“信任底层，智能等待”**。

*   **核心逻辑**：
    既然模组会自动拨号，我们不需要去“教”它怎么拨号，只需要**等待它拨号成功**。
    
*   **关键接口 `checkNet.waitNetworkReady(timeout)`**：
    这是一个官方封装的高级接口，它内部是一个状态机，会自动处理：
    *   阶段 1：等待 SIM 卡就绪。
    *   阶段 2：等待注册到基站（注网）。
    *   阶段 3：等待 PDP 激活（获取 IP）。
    
    只有当这三步都成功，它才会返回 `(3, 1)`。

### 3.3 总结：目前的配置
*   **配置方式**：**开机自动激活**（使用模组默认配置或之前保存的配置）。
*   **代码行为**：**阻塞等待**。代码运行到 `net_mgr.connect()` 时，会暂停执行，直到 `checkNet` 确认网络完全通畅，才继续往下跑。

## 4. 完整的联网流程图解

这个流程展示了从上电到最终能发 HTTP 请求的全过程。

### 4.1 接口对应关系

| 步骤 | 涉及硬件/协议 | QuecPython 接口 | 说明 |
| :--- | :--- | :--- | :--- |
| **1. 检卡** | SIM 卡槽 | `sim.getStatus()` | 检查 SIM 卡是否插入且可读 |
| **2. 注网** | 射频 (RF) | `net.getState()` | 搜索基站信号，注册网络 (CS/PS域) |
| **3. 拨号** | 核心网 (EPC) | `dataCall.activate()` | 建立数据链路，分配 IP |
| **4. 等待** | 软件层 | `checkNet.waitNetworkReady()` | **我们主要调用的接口**，封装了上述 1-3 |
| **5. 传输** | TCP/IP 协议栈 | `usocket.socket()` | 建立 Socket 连接，发送数据 |

## 5. 为什么之前会失败？
之前尝试手动激活时，可能正好撞上了模组后台自动拨号的时间窗口，导致：
*   **资源冲突**：后台正在激活 Profile 1，前台又发指令去激活 Profile 1。
*   **状态不一致**：手动设置 APN 后，没有重启，导致参数未生效，但又强行激活。

现在的方案避开了所有这些坑，直接利用系统最稳定的自动机制。
